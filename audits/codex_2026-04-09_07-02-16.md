# Repository Audit

Date: 2026-04-09 07:02:16
Scope: public, production-ready audit of the foundation toolkit, with special attention to root stack-agnosticity, overlays, docs, prompts, skills, templates, examples, and public-release hygiene.

## Executive Summary

This repository is still **not ready** for a public release as a foundation toolkit.

The main blockers are unchanged and now clearer:
- `docs/tree-manifest.txt` is stale and does not match the actual tree
- root-level Mobile Flutter pack artifacts violate the foundation root contract
- the public docs still read Flutter-first rather than overlay-neutral

The second pass also exposed a hidden packaging artifact: `docs/tree-manifest.mobile-flutter.append.txt`. That file is not referenced by the public workflow and appears to be leftover packaging state. It is not the primary blocker, but it is a sign that the release process is leaking intermediate artifacts into the repo.

## Critical Issues

1. `docs/tree-manifest.txt` is not authoritative
- The public gate fails on tree drift.
- Evidence: `scripts/check-public-repo.sh:32-39`, `scripts/check-public-repo.sh:109-113`
- Evidence: `docs/public-repo-checklist.md:55-63`
- The current drift includes:
  - root files present in the repo but absent from the manifest, including `HOW_TO_APPLY.md` and `VALIDATION.md`
  - the newly added audit artifact in `audits/`
  - the overlay-level `HOW_TO_APPLY.md` and `VALIDATION.md`
  - the entire `overlays/mobile-flutter/skills/**` file subtree, which the manifest only partially represents
  - the stray `docs/tree-manifest.mobile-flutter.append.txt` helper file
- Impact: the repository cannot honestly claim its machine-readable tree snapshot is current, so the public-release gate is not trustworthy.

2. Root-level stack-specific artifacts violate the foundation identity
- `HOW_TO_APPLY.md` and `VALIDATION.md` at the repository root are Mobile Flutter pack artifacts, not foundation artifacts.
- Evidence: `HOW_TO_APPLY.md:1-10`
- Evidence: `VALIDATION.md:1-8`
- The root contract says the foundation must stay stack-agnostic and stack-specific assumptions belong in overlays.
- Evidence: `docs/architecture.md:6-18`
- Impact: a public consumer sees stack-specific packaging material at the root of a repo that claims to be domain-agnostic. That is a direct identity mismatch.

## Medium Issues

1. The navigation layer is still Flutter-biased
- `README.md`, `README_TH.md`, `README_START_HERE.md`, `docs/how-to-use.md`, `docs/how-to-use_TH.md`, `docs/tutorial.md`, `docs/tutorial_TH.md`, `docs/overlays.md`, `docs/tutorials/index.md`, and `docs/tutorials/index_EN.md` all center the Mobile Flutter overlay first.
- Evidence: `README.md:25-69`, `README_TH.md:23-38`, `README_START_HERE.md:9-20`, `docs/how-to-use.md:12-43`, `docs/how-to-use_TH.md:12-41`, `docs/tutorial.md:14-25`, `docs/tutorial_TH.md:14-42`, `docs/overlays.md:12-25`, `docs/tutorials/index.md:3-17`, `docs/tutorials/index_EN.md:3-17`
- Impact: the repository reads like a Flutter-first toolkit with other overlays attached, which weakens the foundation message for backend-node, python-service, and web-frontend adopters.

2. The overlay catalog count is duplicated too widely
- The “23 skills” count appears in several public docs instead of in one overlay-owned place.
- Evidence: `README.md:25-27`, `README_TH.md:23-25`, `docs/how-to-use.md:35-43`, `docs/how-to-use_TH.md:35-41`, `docs/overlays.md:12-25`
- Impact: every future catalog change requires synchronized edits in multiple foundation docs, which increases drift risk for no real user benefit.

3. `docs/tree-manifest.mobile-flutter.append.txt` is stray packaging residue
- The file exists, but nothing in the public docs or scripts references it.
- Evidence: no references found in the repo search outside the file itself.
- Impact: it suggests a hidden packaging workflow and makes the public surface harder to reason about. It is especially confusing because the authoritative manifest is supposed to be single-source truth.

## Minor Issues

1. The tutorial hubs are coherent but not neutral
- `docs/tutorials/index.md` and `docs/tutorials/index_EN.md` still feel like a Flutter launchpad instead of a neutral cross-overlay index.
- Evidence: `docs/tutorials/index.md:3-17`, `docs/tutorials/index_EN.md:3-17`
- Impact: this is not a blocker, but it reinforces the same identity imbalance seen in the root docs.

2. The overlay-level `HOW_TO_APPLY.md` and `VALIDATION.md` are very thin
- These files are overlay-appropriate in principle, but they are almost empty and duplicate the root pack naming.
- Evidence: `overlays/mobile-flutter/HOW_TO_APPLY.md:1-6`, `overlays/mobile-flutter/VALIDATION.md:1-6`
- Impact: they are easy to miss and easy to confuse with the root pack files. If retained, they need clearer positioning or a better naming scheme.

## Foundation vs Overlay Violations

- Root-level Mobile Flutter packaging docs are the clearest violation: `HOW_TO_APPLY.md` and `VALIDATION.md`.
- The root tutorial and entry docs still make Flutter feel like the default rather than one overlay among several.
- `docs/tree-manifest.txt` is itself a boundary issue because the repository is claiming a truthful public tree snapshot while shipping a stale one.
- The overlay-level packaging helpers are not boundary violations in themselves, but they are too thin and too easy to confuse with the root artifacts.

## Public Repo Readiness Assessment

Current status: not ready.

Reasons:
- the tree manifest is not truthful
- the root still contains stack-specific packaging artifacts
- the public docs still tilt toward Flutter instead of a neutral foundation

What is in decent shape:
- public hygiene files are present
- the main docs structure is coherent
- the markdown links I checked resolved
- overlay skill folders appear structurally complete

## Exact Recommended Edits

1. Regenerate `docs/tree-manifest.txt`
- Use the actual repository tree as the source of truth.
- Include the current audit artifact and the overlay skill file subtree.
- Remove the stale append-manifest workflow unless it is intentionally part of the canonical process.

2. Remove or relocate the root Flutter pack artifacts
- Move `HOW_TO_APPLY.md` and `VALIDATION.md` out of the foundation root.
- If they are meant to remain, place them under the overlay or in an external packaging area, not at the root.

3. Rebalance the public navigation
- Update `README.md`, `README_TH.md`, `README_START_HERE.md`, `docs/how-to-use.md`, `docs/how-to-use_TH.md`, `docs/tutorial.md`, `docs/tutorial_TH.md`, and the tutorial hubs so the foundation and all overlays are surfaced more evenly.
- Keep Mobile Flutter visible, but not dominant.

4. Consolidate or reduce the repeated overlay count copy
- Keep the skill-count summary in one place only.
- The rest of the foundation docs should point readers to the catalog instead of repeating the count.

5. Clarify the overlay-level packaging helpers
- If `overlays/mobile-flutter/HOW_TO_APPLY.md` and `overlays/mobile-flutter/VALIDATION.md` stay, add a sentence that explains exactly when they should be used and how they differ from the root-level pack artifacts.
