# Repository Audit

## Executive Summary

- This repository is not ready for a public release tag yet.
- I found 3 critical issues, 4 medium issues, and 4 minor issues.
- The biggest problems are release-contract drift, broken onboarding links, and foundation docs that still default to Flutter in places that should stay stack-agnostic.
- The canonical lifecycle and role docs are broadly aligned, but the public-release gate is too shallow to catch the problems that matter most.

## Critical Issues

1. Tree manifest drift makes the "authoritative" tree snapshot unreliable.
   - `docs/tree-manifest.txt:1-2` claims to be the reference tree snapshot, but the on-disk repo contains many files not listed there.
   - The missing items include `AGENTS.internal.md`, `README_START_HERE.md`, `README_INTERNAL_START_HERE.md`, `PATCH_NOTES.md`, `docs/doc-map.md`, `docs/internal/*`, `prompts/teams/*`, and `scripts/prompt_runner/*`.
   - Impact: the release contract cannot be trusted, and any automation or audit that depends on the manifest is already stale.

2. Public onboarding entrypoints contain broken and internal-only references.
   - `README_START_HERE.md:19-26` points to `tutorials/01_getting_started`, `tutorials/05_reuse_toolkit`, and `tutorials/07_multi_agent`, which do not match the actual file names in the repo.
   - `README_INTERNAL_START_HERE.md:5-11` tells readers to install `AGENTS.internal.md` and copy `prompts/teams/`, but those paths are not represented in the public tree manifest and are not part of the public-facing contract.
   - Impact: first-time users hit stale navigation immediately, and internal-only references leak into root-level docs.

3. Foundation tutorials still default to Flutter in root-level guidance.
   - `docs/tutorials/01-golden-path.md:24-31` and `docs/tutorials/01-golden-path_EN.md:24-31` recommend a Flutter app skeleton as the default first outcome.
   - `docs/tutorials/01-golden-path.md:43-73` and the English equivalent bake a Flutter `AGENTS.md` starter into the foundation tutorial.
   - `docs/tutorials/03-real-workflow.md:48-65`, `docs/tutorials/03-real-workflow_EN.md:48-65`, `docs/tutorials/04-build-production-feature.md:14-55`, and `docs/tutorials/04-build-production-feature_EN.md:14-55` all use a Flutter login feature as the default worked example.
   - Impact: the root tutorial path no longer reads as stack-agnostic foundation guidance; it reads like a Flutter-first starter pack.

## Medium Issues

1. The public-release gate is too shallow to catch the real defects.
   - `scripts/check-public-repo.sh:15-44` only checks file existence and a handful of substrings.
   - It does not validate broken links, tree-manifest parity, or internal-file references in public docs.
   - `docs/public-repo-checklist.md:54-59` says the final checks should confirm the root stays stack-agnostic and the foundation examples stay neutral, but the automation does not actually enforce that.
   - Impact: the script passed during this audit while obvious public-release issues remained.

2. The docs set repeats the same adoption story in too many places.
   - `README.md`, `docs/how-to-use.md`, `docs/real-world-integration.md`, `docs/tutorial.md`, `docs/tutorials/index.md`, and `docs/tutorials/agents-and-prompts.md` all restate the same operating model.
   - This is not wrong by itself, but it is already producing drift: some pages describe the toolkit as generic while others immediately pull readers into Flutter-specific examples.
   - Impact: duplicated guidance increases maintenance cost and makes inconsistent updates more likely.

3. `docs/doc-map.md` is an internal navigation page that does not fit the public contract.
   - `docs/doc-map.md:3-25` points readers at `README_INTERNAL_START_HERE.md`, `docs/internal/*`, `AGENTS.internal.md`, and `prompts/teams/*`.
   - Impact: this is a second, shadow navigation path that bypasses the public docs hierarchy and reinforces the internal/public split inside a public repo.

4. Overlay READMEs still contain copy-pasted wording that is too generic for the stack.
   - `overlays/backend-node/README.md:19-23` includes "top-level UI" language even though the overlay is for backend services.
   - `overlays/python-service/README.md:23-29` is better scoped, but still stays generic enough that it reads like a template rather than a backend-service-specific contract.
   - Impact: the overlay docs are usable, but they do not feel tightly tailored to their own stack.

## Minor Issues

1. `templates/project-bootstrap/README_BOOTSTRAP.md:3-18` is useful but vague about how to choose an overlay.
   - It says "one overlay if relevant" without giving a clear decision rule.

2. `templates/review/review-template.md:1-24` and `templates/verification/verification-template.md:1-24` are structurally sound but do not force line-level evidence.
   - That makes them easy to fill in with vague review notes.

3. `docs/roadmap.md:7-13` prioritizes `mobile-flutter` expansion first without explaining why that stack should lead.
   - It is not a foundation violation by itself, but it reinforces the same Flutter bias seen elsewhere.

4. `docs/tutorials/agents-and-prompts.md:115-141` and the English equivalent still lead with platform-specific starter snippets.
   - The page does call out the stack tutorials as the place for concrete examples, but the opening examples still make the general setup guide feel stack-biased.

## Foundation vs Overlay Violations

- `docs/tutorials/01-golden-path.md` and `_EN.md` push Flutter as the default foundation example.
- `docs/tutorials/03-real-workflow.md` and `_EN.md` use a Flutter login task in a general workflow guide.
- `docs/tutorials/04-build-production-feature.md` and `_EN.md` do the same for a production feature walkthrough.
- `README_START_HERE.md` points new-project readers toward non-existent tutorial slugs instead of the documented canonical path.
- `README_INTERNAL_START_HERE.md` and `docs/doc-map.md` reference internal-only files from public-facing locations.
- `docs/tree-manifest.txt` does not describe the actual tree, so the foundation/overlay boundary cannot be audited reliably from the stated source of truth.

## Public Repo Readiness Assessment

- Status: not ready.
- The repository passes `bash scripts/check-public-repo.sh`, but that gate is insufficient for public-release confidence.
- The manifest is stale, onboarding links are broken, and the root tutorials still bias readers toward Flutter.
- A public maintainer would be handing users a toolkit that advertises stack neutrality while the top-level tutorial path contradicts that claim.
- The repo is close in terms of canonical docs, but not close enough in release hygiene.

## Exact Recommended Edits

1. Regenerate or reconcile `docs/tree-manifest.txt` against the actual repository tree.
   - Decide whether `AGENTS.internal.md`, `docs/internal/*`, `prompts/teams/*`, `scripts/prompt_runner/*`, and `PATCH_NOTES.md` are intentionally public.
   - If they are public, add them to the manifest and public checklist.
   - If they are internal, move them out of the public release surface.

2. Fix `README_START_HERE.md` so every path resolves.
   - Replace `tutorials/01_getting_started` with the actual tutorial file name.
   - Replace `tutorials/05_reuse_toolkit` and `tutorials/07_multi_agent` with the actual tutorial slugs if they differ.
   - Prefer linking to the canonical docs under `docs/tutorials/`.

3. Rework `README_INTERNAL_START_HERE.md` and `docs/doc-map.md`.
   - Either move them into a clearly internal docs area or rewrite them to reference only public files.
   - Remove references to `AGENTS.internal.md` and `prompts/teams/*` from public navigation unless those files are added to the public contract on purpose.

4. Rewrite the root tutorials to be stack-neutral.
   - `docs/tutorials/01-golden-path.md` and `_EN.md` should use a neutral example, such as a generic feature or account preferences task.
   - Move the Flutter-specific starter and `AGENTS.md` example into `docs/tutorials/flutter/`.
   - Apply the same pattern to `docs/tutorials/03-real-workflow*.md` and `docs/tutorials/04-build-production-feature*.md`.

5. Strengthen the public-release gate.
   - Extend `scripts/check-public-repo.sh` to validate link targets in the key public docs.
   - Add a manifest-parity check so tree drift fails the gate.
   - Add an explicit check for internal-path references in public-facing docs.

6. Tighten the overlay README language.
   - Remove the backend-only "top-level UI" wording from `overlays/backend-node/README.md`.
   - Make the overlay descriptions more stack-specific and less template-like.

7. Reduce duplicated guidance.
   - Consolidate the shared adoption narrative into one canonical path, then link to it from the README, how-to docs, and tutorials instead of restating it in full.
   - Keep translations aligned, but do not duplicate the same long-form explanation in five different places if one source of truth will do.

## Final Verdict

- not ready
