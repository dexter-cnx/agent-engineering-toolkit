# Repository Audit: agent-engineering-toolkit

Assessment date: 2026-04-08 16:40:11

## 1. Executive Summary

The repository is structurally strong and the public-repo gate passes, but it is not fully clean as a public foundation toolkit yet.

I found 0 critical issues, 5 medium issues, and 3 minor issues.

The main problems are not missing files. They are consistency and boundary problems:
- the public-repo checklist claims an exact mirror that is not actually exact
- the strict audit prompt omits large parts of the repo surface
- several tutorial and hub pages repeat stack-specific guidance in foundation-owned docs
- the release gate checks presence, not content integrity

Overall verdict: almost ready.

## 2. Critical Issues

None found.

## 3. Medium Issues

1. `docs/public-repo-checklist.md` claims the checklist mirrors the automation "exactly", but the script covers a broader surface than the human list.
   - Evidence: `docs/public-repo-checklist.md:3-6`, `scripts/check-public-repo.paths:1-39`
   - Impact: maintainers may trust the human checklist as exhaustive when it is not, which weakens public-release hygiene and creates false confidence.
   - Recommended edit: rewrite the wording so the checklist is clearly a summary, or expand it until it truly matches the machine-readable source.

2. `docs/strict-audit-prompt.md` under-scopes the repo audit.
   - Evidence: `docs/strict-audit-prompt.md:16-42`
   - Impact: a strict audit run from this prompt can miss `skills/*`, `agent_team/*`, `core/*`, `project_memory/*`, `.github/*`, and `scripts/*`, even though those are part of the public toolkit surface.
   - Recommended edit: widen the audit scope to cover all public repository families, or narrow the claim so the prompt no longer says it is auditing the repository broadly.

3. `docs/tutorials/agents-and-prompts.md` and `docs/tutorials/agents-and-prompts_EN.md` embed full stack-specific AGENTS.md templates in a foundation-owned tutorial hub.
   - Evidence: `docs/tutorials/agents-and-prompts.md:97-160`, `docs/tutorials/agents-and-prompts_EN.md:97-160`
   - Impact: the docs duplicate overlay guidance and make stack-specific defaults feel like foundation guidance, which weakens the root-vs-overlay boundary.
   - Recommended edit: move the concrete Flutter/Web/Services templates into the corresponding stack tutorials, and keep this file as a neutral selector and setup guide.

4. `docs/obsidian-friendly.md` is a second broad source of truth instead of a compact index.
   - Evidence: `docs/obsidian-friendly.md:15-230`
   - Impact: it duplicates the canonical docs, prompt hub, skills list, overlay guidance, and scripts in one large page. That increases drift risk and makes future updates easy to miss.
   - Recommended edit: trim it to a lightweight map of canonical sources, or explicitly mark it as secondary reference only and remove repeated guidance tables.

5. `scripts/check-public-repo.sh` is only an existence gate, so it can pass even when a required file is empty, malformed, or stale.
   - Evidence: `scripts/check-public-repo.sh:13-21`, `docs/public-repo-checklist.md:3-6`
   - Impact: the release workflow can still produce a green check while the public-facing content is structurally present but not actually healthy.
   - Recommended edit: add a few content-level sanity checks for the canonical docs and GitHub files, or document very clearly that this is only a presence gate and is not a content validator.

## 4. Minor Issues

1. The overlay README and overlay AGENTS files repeat the same boundary rules twice.
   - Evidence: `overlays/mobile-flutter/README.md:18-38`, `overlays/mobile-flutter/AGENTS.overlay.md:3-21`, and the same pattern in `overlays/backend-node/`, `overlays/web-frontend/`, and `overlays/python-service/`
   - Impact: the duplication is usable as a quick reference, but it increases drift risk when one copy changes and the other does not.
   - Recommended edit: keep one canonical location per overlay for the rules, and make the other file point at it more explicitly.

2. The review and verification templates do not explicitly ask for foundation-vs-overlay or public-release checks.
   - Evidence: `templates/review/review-template.md:3-31`, `templates/verification/verification-template.md:3-29`
   - Impact: reviewers can complete the templates without explicitly checking the repo’s core identity constraints, which are central to this toolkit.
   - Recommended edit: add one line item to each template for foundation-vs-overlay compliance and public-release hygiene when relevant.

3. The prompt aliases at the root are minimal and technically correct, but they create an extra layer of indirection with no added guidance.
   - Evidence: `prompts/audit_repo.md:1-3`, `prompts/plan_change.md:1-3`, `prompts/implement_change.md:1-3`, `prompts/review_change.md:1-3`, `prompts/verification_pass.md:1-3`, `prompts/finalize_change.md:1-3`
   - Impact: the aliases are fine for compatibility, but they do not help adoption and can confuse readers who do not realize the canonical files live deeper in the tree.
   - Recommended edit: keep them, but add one short root-level prompt hub note that explains they are compatibility shims only.

## 7. Public Repo Readiness Assessment

The public repository is close, but not fully release-clean.

What is ready:
- the required root files exist
- GitHub-facing hygiene files exist
- canonical lifecycle and role docs are present
- overlays are separated from the foundation
- the public-repo gate passes

What is still weak:
- the public checklist overclaims exact coverage
- the strict audit prompt misses large parts of the repo
- the documentation surface repeats itself in multiple hubs
- the release gate checks presence, not file quality

Net assessment: the repo is publishable with attention, but it is not yet polished enough to call fully ready without caveats.
