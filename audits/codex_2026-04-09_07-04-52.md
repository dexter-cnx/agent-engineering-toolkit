# Repository Audit

Date: 2026-04-09 07:04:52
Scope: public, production-ready audit of the foundation toolkit, with special attention to root stack-agnosticity, overlays, docs, prompts, skills, templates, examples, and public-release hygiene.

## Executive Summary

This repository is **not ready** for a public release as a foundation toolkit.

The most important current blocker is that `docs/tree-manifest.txt` is not authoritative. The public gate fails because the manifest and the real tree still disagree.

Two important state notes:
- the root-level `HOW_TO_APPLY.md` and `VALIDATION.md` are now deleted in the working tree, which is the right direction for a foundation root
- `docs/tree-manifest.mobile-flutter.append.txt` still exists and is not referenced by the public workflow, so the packaging story is still leaking intermediate artifacts

The foundation docs also still lean Flutter-first in several places, so the root navigation is still not neutral enough for a public multi-overlay toolkit.

## Critical Issues

1. `docs/tree-manifest.txt` is still not authoritative
- The public-release gate fails on tree drift.
- Evidence: `scripts/check-public-repo.sh:32-39`, `scripts/check-public-repo.sh:109-113`
- Evidence: `docs/public-repo-checklist.md:55-63`
- The current drift includes:
  - the newly created audit artifacts under `audits/`
  - the overlay-level `HOW_TO_APPLY.md` and `VALIDATION.md`
  - the stray `docs/tree-manifest.mobile-flutter.append.txt` helper file
  - many overlay skill files and directories that are not represented with full fidelity in the manifest snapshot
- Impact: the repository cannot honestly claim its machine-readable tree snapshot is current, so the public-release gate is unreliable.

## Medium Issues

1. The foundation navigation is still Flutter-biased
- `README.md`, `README_TH.md`, `README_START_HERE.md`, `docs/how-to-use.md`, `docs/how-to-use_TH.md`, `docs/tutorial.md`, `docs/tutorial_TH.md`, `docs/overlays.md`, `docs/tutorials/index.md`, and `docs/tutorials/index_EN.md` still center the Mobile Flutter overlay first.
- Evidence: `README.md:25-69`, `README_TH.md:23-38`, `README_START_HERE.md:9-20`, `docs/how-to-use.md:12-43`, `docs/how-to-use_TH.md:12-41`, `docs/tutorial.md:14-25`, `docs/tutorial_TH.md:14-42`, `docs/overlays.md:12-25`, `docs/tutorials/index.md:3-17`, `docs/tutorials/index_EN.md:3-17`
- Impact: the repository still reads like a Flutter-first toolkit with other overlays attached, which weakens the foundation message for backend-node, python-service, and web-frontend adopters.

2. The overlay catalog count is duplicated too widely
- The “23 skills” count appears in several public docs instead of in one overlay-owned place.
- Evidence: `README.md:25-27`, `README_TH.md:23-25`, `docs/how-to-use.md:35-43`, `docs/how-to-use_TH.md:35-41`, `docs/overlays.md:12-25`
- Impact: every future catalog change requires synchronized edits in multiple foundation docs, which increases drift risk for no real user benefit.

3. `docs/tree-manifest.mobile-flutter.append.txt` is stray packaging residue
- The file exists, but nothing in the public docs or scripts references it.
- Evidence: repo search found no references outside the file itself.
- Impact: it suggests a hidden packaging workflow and makes the public surface harder to reason about. That is especially confusing because the canonical manifest is supposed to be the single source of truth.

4. The overlay-level `HOW_TO_APPLY.md` and `VALIDATION.md` are too thin
- These files exist under `overlays/mobile-flutter/`, but they are extremely minimal.
- Evidence: `overlays/mobile-flutter/HOW_TO_APPLY.md:1-6`, `overlays/mobile-flutter/VALIDATION.md:1-6`
- Impact: they are easy to miss and easy to confuse with root-level packaging files. If retained, they need clearer naming or stronger guidance about their limited purpose.

## Minor Issues

- `docs/tutorials/index.md` and `docs/tutorials/index_EN.md` still feel like a Flutter launchpad instead of a neutral cross-overlay index.
- Evidence: `docs/tutorials/index.md:3-17`, `docs/tutorials/index_EN.md:3-17`
- Impact: this is not a blocker, but it reinforces the same identity imbalance seen in the root docs.

2. The root pack deletion is not yet reflected in the public release contract
- `HOW_TO_APPLY.md` and `VALIDATION.md` are deleted in the working tree, which is good for the foundation identity.
- Evidence: `git status --short` shows them deleted; the files are absent from the workspace.
- Impact: the working tree is still not release-ready until the manifest and release gate are updated to match the new state.

## Exact Recommended Edits

1. Regenerate `docs/tree-manifest.txt`
- Use the actual repository tree as the source of truth.
- Include the current audit artifact and the overlay skill file subtree.
- Remove the stale append-manifest workflow unless it is intentionally part of the canonical process.

2. Decide what to do with `docs/tree-manifest.mobile-flutter.append.txt`
- If it is part of a real packaging workflow, document and integrate it.
- If it is only leftover packaging residue, delete it.

3. Rebalance the public navigation
- Update `README.md`, `README_TH.md`, `README_START_HERE.md`, `docs/how-to-use.md`, `docs/how-to-use_TH.md`, `docs/tutorial.md`, `docs/tutorial_TH.md`, and the tutorial hubs so the foundation and all overlays are surfaced more evenly.
- Keep Mobile Flutter visible, but not dominant.

4. Consolidate or reduce the repeated overlay count copy
- Keep the skill-count summary in one place only.
- The rest of the foundation docs should point readers to the catalog instead of repeating the count.

5. Clarify the overlay-level packaging helpers
- If `overlays/mobile-flutter/HOW_TO_APPLY.md` and `overlays/mobile-flutter/VALIDATION.md` stay, add a sentence that explains exactly when they should be used and how they differ from the old root-level pack artifacts.
