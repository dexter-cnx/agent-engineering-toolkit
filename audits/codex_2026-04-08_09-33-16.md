# Codex Audit Report

Repository identity:
- domain-agnostic foundation toolkit
- stack-agnostic at root
- stack-specific assumptions in overlays only

Audit scope:
- README.md
- README_TH.md
- AGENTS.md
- docs/*
- overlays/*
- templates/*
- examples/*
- core/*
- skills/*
- public repo hygiene files

## Executive Summary

- Foundation-vs-overlay boundaries are mostly clean.
- The main blocker is public-repo readiness: the repo claims `.github/` and CI/public-hygiene files, but they are missing.
- Documentation drift is the next biggest risk. There are two competing how-to guide pairs, and the lifecycle is repeated in many places.
- Skills are mostly narrow, but several are too minimal to be operationally reusable without additional contract detail.

## Critical Issues

1. Public repo hygiene is incomplete.
   - `README.md` and `README_TH.md` claim `.github/` and CI exist.
   - The repository has no `.github/` directory, no workflows, no `CODEOWNERS`, no `SECURITY.md`, no PR template, and no issue templates.
   - This is a release blocker for a public toolkit repo.

2. Public release docs overstate the current state.
   - `docs/release-v0.1.0.md` lists CI sanity checks and public repository hygiene files as included.
   - That is not true in the current tree.

## Medium Issues

1. Guide-path drift.
   - `README.md` points to `docs/how-to-use.md` and `docs/how-to-use_TH.md`.
   - `docs/tutorial.md` and `docs/tutorial_TH.md` point to `docs/how_to_use.md` and `docs/how_to_use_TH.md`.
   - Both underscore files are longer and contain extra material, so the repo currently has two competing guide paths.

2. Prompt pipeline duplication.
   - The same lifecycle appears in `README.md`, `docs/prompt-pipeline.md`, `docs/how-to-use.md`, `docs/how_to_use.md`, `templates/runbooks/feature-flow.md`, and `examples/workflows/feature-workflow.md`.
   - This is likely to drift unless one canonical source is declared.

3. Templates are too skeletal for public adoption.
   - `templates/project-bootstrap/README_BOOTSTRAP.md`
   - `templates/review/review-template.md`
   - `templates/verification/verification-template.md`
   - These are placeholders, not immediately reusable operating templates.

4. Example feature prompt is incomplete.
   - `examples/prompts/example_feature_request.md` omits `implement_change.md`, `finalize_change.md`, and `update_project_memory.md`.
   - That weakens the canonical lifecycle message.

5. Skill contracts are underspecified.
   - Several skill READMEs only contain a purpose line.
   - They need input/output boundaries and non-goals to be reusable in real workflows.

## Minor Issues

1. `agent_team/README.md` is too sparse to be useful on its own.
2. `docs/public-repo-checklist.md` should name exact files instead of generic placeholders.
3. `skills/repo-audit/checklist.md` is too short to support the audit scope implied by the repo.
4. `docs/overlays.md` is fine, but should stay strictly descriptive and avoid becoming policy.

- `README_TH.md`: matches English README, same release-hygiene gap.
- `AGENTS.md`: clean and appropriately stack-agnostic.

- Skills are narrow by intent, but many need explicit contract sections to be reusable.

## Foundation vs Overlay Violations

- No strong root-level violation found that makes Flutter, Node, or Python the default toolkit identity.
- Stack references in the foundation docs are mostly used for overlay selection, which is acceptable.
- Overlay READMEs and overlay AGENTS files extend the foundation instead of rewriting it.

## Public Repo Readiness Assessment

- Not ready.
- Missing `.github/` content and public-hygiene files are the main blocker.
- Documentation duplication and guide-path drift should be cleaned before a public release.
