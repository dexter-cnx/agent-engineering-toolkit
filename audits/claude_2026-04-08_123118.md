# Audit: agent-engineering-toolkit
**Auditor:** Claude (claude-sonnet-4-6)
**Date:** 2026-04-08
**Scope:** Public, production-ready foundation toolkit audit
**Branch audited:** develop

---

## 1. Executive Summary

The repository has a coherent, well-reasoned identity: a domain-agnostic foundation toolkit for structured AI-assisted engineering. The foundational architecture (foundation/overlay separation, canonical lifecycle, role model) is sound and consistently enforced across root-level files. The primary docs and 7 canonical lifecycle prompts are production-quality.

However, the repo is **not ready** as a public production foundation. The specific blockers:

- **CI will fail on every push** — a file path in `ci.yml` references a file that does not exist at that path.
- **CHANGELOG version numbering is incoherent** — all versions from 0.1.0 to 1.0.0 are dated the same day, and 0.1.1 is missing entirely.
- **Four prompts (`audit_repo`, `scaffold_overlay`, `investigate_bug`, `adopt_toolkit_in_repo`) are skeletal** and inconsistent with the structured format of the main 7 lifecycle prompts.
- **Three of four overlays are significantly weaker** than the Python overlay and cannot credibly support adoption.
- **Two scripts do not function as implied** — they echo commands rather than execute them.
- **Internal operational docs are publicly exposed** — `docs/push-guide.md` and old release notes in `docs/` are visible to adopters with no navigation purpose.
- **`project_memory/patterns.md` is missing** despite the template existing.

These are fixable. The core framework design is correct. The gap is between the high-quality skeleton and the lower-quality filler around it.

## 2. Critical Issues

### C1 — CI WILL FAIL: Wrong file path in `ci.yml`

**File:** `.github/workflows/ci.yml`, line 34

**Problem:**
```yaml
test -f examples/worked_examples/python_service_feature.md
```

This file does **not exist** at `examples/worked_examples/python_service_feature.md`.
The actual file is at `overlays/python-service/examples/python_service_feature.md`.

This causes CI to fail on every push. It is a release blocker.

**Fix:**
```yaml
# Replace line 34 with:
test -f overlays/python-service/examples/python_service_feature.md
```

### C2 — CHANGELOG version numbering is incoherent

**File:** `CHANGELOG.md`

**Problem:**
All versions — 0.1.0, 0.1.2, 0.1.3, and 1.0.0 — share the date `2026-04-08`. Version 0.1.1 is missing with no explanation. A same-day jump from 0.1.0 to 1.0.0 signals the changelog was constructed retroactively in a single session and does not reflect real release history. External contributors will question the project's release discipline immediately.

**Fix options:**
1. If only one release has actually happened, consolidate into a single `[0.1.0] - 2026-04-08` entry with cumulative changelog content.
2. If versions reflect real iteration milestones, add a note at the top of CHANGELOG explaining that early versions were developed in rapid succession and do not reflect calendar time.
3. At minimum, add a note explaining the missing 0.1.1 and the same-day versioning.

Leaving it as-is gives the repo a manufactured release history, which undermines credibility.

## 3. Medium Issues

### M1 — Four prompts are skeletal, inconsistent with the quality of the 7 canonical lifecycle prompts

**Files:**
- `prompts/audit_repo.md` — 6 bullet points, no role, no process, no required output
- `prompts/scaffold_overlay.md` — 5 bullet points, no role, no process, no required output
- `prompts/investigate_bug.md` — closer to the right shape but still lacks Inputs, Required output, Non-goals sections
- `prompts/adopt_toolkit_in_repo.md` — 4 focus areas, no role, no process, no required output

**Problem:** The 7 lifecycle prompts (`plan_change.md` through `update_project_memory.md`) all have: Role, Goal, When to use, Inputs, Process (numbered steps), Required output, Non-goals. The 4 non-lifecycle prompts have none of this structure. A user reading the toolkit gets an inconsistent experience — four prompts look like abandoned stubs next to seven production-quality ones. This is an adoption friction risk and a quality signal problem for a public release.

**Fix:** Bring all four prompts up to the same structural standard. At minimum:
- Add `## Role`, `## Goal`, `## When to use`, `## Inputs`, `## Process`, `## Required output`, `## Non-goals` to each.
- If the prompts are intentionally "light-touch," add an explicit note explaining that, but they still need more content than 4-6 bullet points.

### M2 — Three overlays are significantly weaker than the Python overlay

**Files:**
- `overlays/mobile-flutter/AGENTS.overlay.md` — 4 rules
- `overlays/mobile-flutter/README.md` — 13 lines, no structure, no examples
- `overlays/backend-node/AGENTS.overlay.md` — 4 rules
- `overlays/backend-node/README.md` — 13 lines, no structure, no examples
- `overlays/web-frontend/AGENTS.overlay.md` — 4 rules
- `overlays/web-frontend/README.md` — 13 lines, no structure, no examples

**Compared to:** `overlays/python-service/` which has: recommended directory structure (with code block), explicit responsibility mapping, verification examples with bash commands, review guidance with rejection criteria, memory guidance, and a worked example.

**Problem:** A team adopting `mobile-flutter` or `backend-node` gets 4 bullet points. A team adopting `python-service` gets a near-complete operational guide. This is a significant quality disparity that makes the non-Python overlays feel like stubs. For a claimed production-ready public toolkit, three out of four overlays failing to meet the standard of the fourth is a structural problem.

**Fix:** For each weak overlay, add at minimum:
- A recommended directory structure (code block)
- Responsibility mapping per directory
- At least two concrete verification command examples
- A short worked example (or reference the foundation example with overlay-specific notes)

### M3 — `agent_team/` role files add no depth over `docs/agent-team-system.md`

**Files:** `agent_team/lead.md`, `architect.md`, `builder.md`, `reviewer.md`, `verifier.md`, `finalizer.md`, `memory.md`

**Problem:** Every role file is 4-5 bullet points that are verbatim summaries of what `docs/agent-team-system.md` already states in full. A user going to `agent_team/lead.md` learns nothing they wouldn't learn from the system doc. The `agent_team/README.md` acknowledges this: "For the detailed explanation of responsibilities and non-overlap rules, read: `docs/agent-team-system.md`." The role files serve as a cross-reference at best, navigation clutter at worst.

**Fix:** Either:
1. Expand each role file with concrete deliverables per lifecycle stage, anti-patterns, handoff criteria, and example outputs — making them genuinely useful standalone references.
2. Or remove the role files entirely and update `agent_team/README.md` to simply point to `docs/agent-team-system.md`.

Current state: they exist and are not harmful, but they do not justify their presence in a production toolkit.

### M4 — `core/` is largely redundant with `docs/`, `AGENTS.md`, and `skills/`

**Files:** `core/rules/foundation-rules.md`, `core/rules/output-contract.md`, `core/routing/skill-router.md`, `core/routing/risk-model.md`, `core/architecture/layering.md`, `core/review/review-checklist.md`, `core/verification/verification-baseline.md`

**Problem:**
- `core/rules/foundation-rules.md` (5 bullets) duplicates `AGENTS.md`
- `core/rules/output-contract.md` duplicates `AGENTS.md`'s mandatory output structure
- `core/routing/skill-router.md` (7 lines) is nearly identical to `skills/skill-router/README.md`
- `core/routing/risk-model.md` (13 lines) overlaps with `skills/risk-scoring/README.md`
- `core/architecture/layering.md` duplicates `docs/architecture.md`
- `core/review/review-checklist.md` (5 lines) is a thin subset of `templates/review/review-template.md`
- `core/verification/verification-baseline.md` (4 lines) is a thin subset of `templates/verification/verification-template.md`

These files imply depth that doesn't exist. They create dual-maintenance risk: if `AGENTS.md` or `docs/architecture.md` evolves, someone must remember to update `core/` as well.

**Fix:** Either give `core/` genuine operational depth (decision tables, expansion of rules, concrete examples of violations) or consolidate into the canonical locations and remove `core/` or make it clearly a "quick-reference index" with explicit links to canonical sources.

### M5 — Scripts do not execute — they echo commands

**Files:** `scripts/adopt-in-project.sh`, `scripts/push-new-repo.sh`

**Problem:**
- `adopt-in-project.sh` does `git submodule add` (actually runs it), creates a directory, then only *echoes* template file paths instead of copying them. A user running this script ends up with a submodule added but no template files copied, despite the echo output suggesting they should do it next.
- `push-new-repo.sh` only echoes a sequence of git commands with `echo "..."` — it does not execute any of them. The script title implies it pushes a repo; it does not.

**Fix:**
- `adopt-in-project.sh`: Replace `echo` lines with actual `cp` commands for template files. Add a fallback note if the toolkit is not a submodule.
- `push-new-repo.sh`: Either rename to `scripts/push-guide.sh` and make it clear it's reference output only, or convert it to an actual executable script with proper error handling.

### M6 — `examples/workflows/feature-workflow.md` and `templates/runbooks/feature-flow.md` are not what they claim to be

**File:** `examples/workflows/feature-workflow.md` — 10 lines, 3 reminder notes. Not a workflow example.
**File:** `templates/runbooks/feature-flow.md` — 4 lines. Not a runbook.

**Problem:** Users who navigate to `examples/workflows/` expect to see how a workflow plays out. Users who open a "runbook" template expect operational guidance. Both files are reminder stubs that add nothing an experienced user doesn't already know and leave new users with nothing actionable.

**Fix:**
- `feature-workflow.md`: Write a brief concrete scenario (e.g., "adding a config endpoint to a generic API service") with inputs, abbreviated stage outputs, and what good vs. poor results look like at each stage.
- `feature-flow.md` runbook: Add concrete operational steps (pre-conditions, go/no-go criteria per stage, escalation path for blocked stages, post-completion checklist).

They document internal development progression and are only relevant to the maintainer. ### M8 — `examples/prompts/example_feature_request.md` provides no learning value

**File:** `examples/prompts/example_feature_request.md`

**Content:**
```
Follow AGENTS.md strictly.
Task: Add an account preferences feature.
Use the canonical lifecycle from docs/prompt-pipeline.md.
[list of 7 prompt files]
```

**Problem:** This is not an example — it's a template invocation without context. A new user reading this learns: (1) the task name, (2) the prompt list (which they already know from the tutorial). There is no expected input shape, no sample output, no quality signals. It does not demonstrate what a good feature request looks like or what the AI is expected to produce at each stage.

**Fix:** Add: scenario context (what is "account preferences"?), expected planning output, expected architecture output summary, and what "good" looks like at review and verification stages. Alternatively, replace with the `foundation_feature_flow.md` scenario and retire this file as redundant.

**File:** `CODE_OF_CONDUCT.md`

**Problem:** "Contact the maintainer via the repository profile or private email configured for the repository." There is no email listed, and "repository profile" is not a contact mechanism. This is unusable as an enforcement path for contributors who encounter a conduct issue.

**Fix:** Add a specific contact method — either a GitHub private advisory link, a dedicated email address, or a direct GitHub profile link.

### M10 — `docs/how-to-use.md` Sections 6 and 7 are tool-specific and largely redundant

**File:** `docs/how-to-use.md`

**Problem:** Section 6 is "Using with Codex" and Section 7 is "Using with Claude Code." Both sections provide nearly identical guidance (read AGENTS.md, use lifecycle, update memory). "Codex" (OpenAI Codex) is a specific product that may confuse users not using it. The toolkit claims to be tool-agnostic but names two specific AI products in the canonical usage guide.

**Fix:** Merge into one section: "## Using with AI coding tools" with a note that the same pattern applies regardless of which AI tool is used. Optionally keep tool-specific notes as sub-subsections if the differences are meaningful (they currently are not).

But `project_memory/patterns.md` does not exist, despite `project_memory/README.md` implying three primary files (decisions, known_constraints) and the template offering three. ### m2 — `docs/tree-manifest.txt` is stale and unexplained

**File:** `docs/tree-manifest.txt`

The manifest does not include the `audits/` directory or any of its files. It is not referenced from README, any guide, or CI. Its purpose is unclear to a public user.

**Fix:** Either update the manifest to reflect the current tree (including `audits/`), add a note at the top explaining it is a reference artifact, and add a CI check to keep it current — or delete it and stop maintaining a separate manifest that drifts from the actual tree.

### m3 — `SECURITY.md` root file is a single-line redirect

**File:** `SECURITY.md`

GitHub auto-surfaces `.github/SECURITY.md` for security policy. The root `SECURITY.md` only says "Canonical security policy: `.github/SECURITY.md`. For the authoritative policy, read `.github/SECURITY.md`." This adds a navigation step with no benefit.

**Fix:** Either delete the root file and rely on GitHub's `.github/SECURITY.md` detection, or copy the policy content into the root so users who find it directly have the full information.

---

### m4 — `docs/roadmap.md` contains no dates, priorities, or criteria

**File:** `docs/roadmap.md`

"Next:" lists 5 items. "Later:" lists 5 more. No dates, no priorities, no "in progress" signals. For a public repo, a roadmap that is indistinguishable from an idea list is not useful for contributors evaluating whether to adopt or contribute.

**Fix:** Add at minimum: which items are actively being worked on, a rough priority ordering, and which items are speculative.

---

### m5 — `.gitignore` is thin for a toolkit with multi-stack audience

**File:** `.gitignore`

Current entries: `.DS_Store`, `*.swp`, `*.bak`, `.idea/`, `.vscode/`

Missing for typical multi-stack environments: `__pycache__/`, `*.pyc`, `.env`, `node_modules/`, `*.log`. Teams consuming this toolkit and running scripts will produce these artifacts.

**Fix:** Add common entries. This does not need to be exhaustive — the consuming repo can extend `.gitignore` — but a public foundation toolkit should model good hygiene.

---

### m6 — CHANGELOG 0.1.1 is missing without explanation

**File:** `CHANGELOG.md`

Versions go 0.1.0 → 0.1.2, skipping 0.1.1. No explanation. Minor for a new repo, but external contributors will notice.

**Fix:** Add a comment in the changelog explaining the version was skipped or consolidate the changelog as noted in C2.

---

### m7 — `.github/SECURITY.md` lacks a disclosure timeline

**File:** `.github/SECURITY.md`

The file correctly says "Use a private GitHub security advisory or the maintainer's private contact channel" but sets no expectations for response time. For a public toolkit, contributors need to know within what window to expect acknowledgment.

**Fix:** Add: "We will acknowledge receipt within 72 hours and provide a remediation timeline within 7 days."

---

### m8 — `docs/overlays.md` step 3 ("Merge overlay assumptions into the consuming repo") is vague

**File:** `docs/overlays.md`, step 3 of "How to use an overlay"

"Merge overlay assumptions into the consuming repo" is ambiguous. Does this mean copy the AGENTS.overlay.md file? Reference it? Extend it? A new user cannot determine what concrete action to take.

**Fix:** Replace with: "Copy the `AGENTS.overlay.md` file into your consuming repository and extend it with project-specific rules. Do not modify the original overlay file — add local rules on top."
