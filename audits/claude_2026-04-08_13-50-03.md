## 4. Minor Issues

### m1 — Worked examples skip Steps 3 and 6 without explanation (adoption confusion)

**Files affected:**
- `overlays/mobile-flutter/examples/worked_example.md`
- `overlays/backend-node/examples/worked_example.md`
- `overlays/web-frontend/examples/worked_example.md`

**Problem:** Each example's section numbering jumps: 1, 2, 4, 5, 7. Steps 3 (Implement) and 6 (Finalize) are absent with no explanation. The lifecycle defines 7 stages. An adopter reading the example while following `docs/prompt-pipeline.md` will notice the missing steps and be uncertain whether to skip them in practice.

The python-service example covers all 7 stages. These three do not and provide no rationale for the omission. This is not a functional block but it is weak adoption guidance — the toolkit explicitly requires verification, finalization, and honest accounting of skipped stages.

**Fix option A (preferred):** Add abbreviated Steps 3 and 6 to each example. Step 3 can be two bullet points ("files changed" + "deviations: none"). Step 6 can be one line ("result is coherent, follow-up: add integration test").

**Fix option B:** Add a note at the top of each example: "Steps 3 (Implement) and 6 (Finalize) are abbreviated in this example. See `overlays/python-service/examples/python_service_feature.md` for a full 7-stage walkthrough."

### m2 — `docs/real-world-integration.md` Section 3 is now vague after the Codex rename

**File:** `docs/real-world-integration.md`

**Problem:** Section 2 is now "AI coding tool workflow" and Section 3 is "Alternate AI coding tool workflow." Both contain essentially the same instruction: use `AGENTS.md` as contract, `docs/prompt-pipeline.md` as lifecycle source, `docs/agent-team-system.md` as role model. The prior version distinguished Section 2 ("Codex workflow") from Section 3 ("Claude Code workflow") — specific tools, meaningful distinction. The rename to make both generic removed the distinction entirely.

A reader cannot determine what "alternate" means or how Sections 2 and 3 differ. The word "alternate" is now vague filler.

**Fix:** Either collapse Sections 2 and 3 into a single "AI coding tool workflow" section (since they say the same thing), or restore meaningful distinction by naming what differs between tool types — e.g., system prompt injection vs. in-conversation context, file-reading capabilities, multi-agent setups.

---

### m3 — `scripts/final-review-prompt.txt` is a duplicate that will silently drift

**File:** `scripts/final-review-prompt.txt`

**Problem:** The file contains the full audit prompt, labeled "This is a convenience copy of the strict audit prompt. Canonical source: `docs/strict-audit-prompt.md`." This is better than the prior one-line stub, but it introduces a two-source sync problem. If `docs/strict-audit-prompt.md` is updated — new audit goals, changed output format, refined rules — `scripts/final-review-prompt.txt` will silently contain the stale version with no visible indication.

The tutorials (EN and TH) reference `scripts/final-review-prompt.txt` as a usable resource. A user following the tutorial and using this file may run an outdated prompt without knowing.

**Fix option A:** Remove `scripts/final-review-prompt.txt` and update the tutorials to reference only `docs/strict-audit-prompt.md`. The tutorials already list both files — removing the convenience copy eliminates the sync burden with no functional loss.

**Fix option B:** Keep the file but add a checksum or "last synced from docs/strict-audit-prompt.md on [date]" comment so drift becomes visible during future audits.

---

### m4 — CHANGELOG same-day multi-version history (carried over from prior audit)

**File:** `CHANGELOG.md`

**Problem:** Unchanged since prior audit. Versions 0.1.0, 0.1.2, 0.1.3, 1.0.1, and 1.0.2 are all dated 2026-04-08. The explanatory note at the top mitigates this but does not eliminate the credibility concern. A first-time external contributor or adopter evaluating the project's maturity will still see a changelog that describes 5 versions shipped in a single day, spanning from 0.1.0 to 1.0.2.

This does not block adoption. It affects external perception of release discipline.

**Fix (low effort):** Collapse all 0.x entries into a single `[0.1.0] - 2026-04-08` entry marked "initial development iteration — not individually tagged" and keep only the 1.x series as the public release history. This is honest and cleaner than 5 same-day entries with a disclaimer.
