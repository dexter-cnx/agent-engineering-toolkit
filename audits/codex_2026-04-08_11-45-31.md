# Codex Audit Report

Repository identity:
- domain-agnostic foundation toolkit
- stack-agnostic at root
- stack-specific assumptions belong in overlays only

Audit scope:
- `README.md`
- `README_TH.md`
- `AGENTS.md`
- `docs/*`
- `overlays/*`
- `templates/*`
- `examples/*`

- public repo hygiene files

## Executive Summary

- The foundation-vs-overlay boundary is now mostly clean.
- The prompt pipeline is coherent and has been centralized on `docs/prompt-pipeline.md`.
- The biggest remaining blocker is public repo hygiene: the repo advertises GitHub-facing files and `.gitignore`, but those files are still missing from the tree.
- The docs/templates/skills surface is much stronger than before, but the public-release story is still incomplete because the repository is describing infrastructure it does not actually contain.

## Critical Issues

1. Public repo hygiene is still incomplete.
   - `README.md:44-58` and `README_TH.md:44-58` claim `.github/` exists for CI, issue templates, PR template, CODEOWNERS, and security policy.
   - `docs/public-repo-checklist.md:10-22` marks those files as required.
   - The tree has no `.github/` directory, no `.gitignore`, and no GitHub-facing public-release files.
   - This blocks public-release readiness because the repo is describing release scaffolding that does not exist.

2. `CHANGELOG.md` overstates the current repository state.
   - It claims "Real `.github/` public-repo scaffolding" and "Canonical lifecycle reference enforcement in CI".
   - Those items are not present in the current tree.
   - For a public toolkit repo, that is a material accuracy problem in release notes.

## Medium Issues

1. `docs/public-repo-checklist.md` is ahead of the tree.
   - The checklist now names the exact files, which is good.
   - But it still describes a target state rather than a current one, so maintainers may assume the scaffold exists when it does not.

2. `docs/push-guide.md` and `scripts/push-new-repo.sh` are still generic examples only.
   - They are fine as examples, but they do not help a maintainer create the missing public-release files.
   - The repo needs an actual bootstrap path for the `.github` scaffold, not only push commands.

3. `docs/release-v0.1.0.md` is now safer, but it is still release-history prose with limited verification context.
   - That is acceptable for a note file.
   - It should not be treated as evidence that the public scaffold exists.

## Minor Issues

1. `docs/how-to-use.md` and `docs/how-to-use_TH.md` are now canonical and aligned, but their advice still relies on the reader discovering related templates manually.
2. `project_memory/README.md` is minimal; it would be more useful if it pointed to the exact memory files and update rules.
3. `agent_team/README.md` remains very sparse compared with the stronger `docs/agent-team-system.md`.
