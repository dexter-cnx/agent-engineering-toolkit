# Repository Audit

## Executive Summary
This repository is close to a credible public foundation toolkit, but it is not yet clean enough to call fully release-ready without caveats.

The strongest parts are the root identity, the canonical lifecycle and role docs, the public-repo hygiene files, and the existence of explicit overlays. The main weaknesses are maintenance drift and adoption clarity: the authoritative tree manifest is stale, the audit prompt is duplicated in two places, and the adoption/tutorial surface repeats itself without giving new users a sharper first-run path.

## Critical Issues
- None found.

## Medium Issues
- `docs/tree-manifest.txt:1-2,4-155` is stale as an "authoritative" tree reference because it omits the tracked `audits/` directory and its audit files. That makes the snapshot inaccurate and weakens the manifest's value as a release/audit reference. Exact fix: add `audits/` and the tracked audit files to the manifest, or explicitly document that audit artifacts are intentionally excluded. The current file does neither.
- `prompts/audit_repo.md:1-45` and `docs/strict-audit-prompt.md:1-58` duplicate the same audit intent and output contract. This creates two near-canonical audit prompts and invites drift. Exact fix: make `prompts/audit_repo.md` a thin pointer to `docs/strict-audit-prompt.md`, or remove one of them.
- `docs/how-to-use.md:48-129`, `docs/tutorial.md:13-80`, `examples/workflows/feature-workflow.md:1-23`, and `examples/workflows/toolkit-adoption.md:1-16` overlap heavily. The adoption path is documented in too many near-equivalent forms, but none of them give a sharply concrete first-run walk-through. Exact fix: keep `docs/how-to-use.md` as the canonical adoption guide, trim the workflow summary files to short pointers, and rewrite `docs/tutorial.md` to show one end-to-end worked run with explicit inputs, outputs, and a memory entry.

## Minor Issues
- `scripts/adopt-in-project.sh:1-24` is named like a full adoption helper but only bootstraps the `project_memory` files after adding the submodule. It does not establish `AGENTS.md`, overlay choice, or verification commands, so the script understates the manual work still required. Exact fix: rename it to reflect the limited scope, or extend it to scaffold the remaining adoption steps.
- `overlays/mobile-flutter/README.md:1-38`, `overlays/backend-node/README.md:1-39`, and `overlays/web-frontend/README.md:1-39` are very similar in structure and wording. They are usable, but the stack-specific differentiation is thin, so the overlay READMEs read more like generic placeholders than opinionated adoption guides. Exact fix: add stack-specific "when to use", "boundary gotchas", and "verification nuance" sections to each overlay README.

## Foundation vs Overlay Violations
- No direct root-level stack-specific leakage was found in `README.md`, `AGENTS.md`, `docs/*`, or the root prompt set.
- The only boundary concern is indirect: the overlay READMEs are so generic that they risk looking like copy-pasted foundation guidance instead of stack-specific extensions.

## Public Repo Readiness Assessment
- Required public files are present.
- Release-check and CI exist.
- The root identity is explicit and stack-agnostic.
- Public-facing governance files are in place.
- The main gap is maintenance truthfulness, not missing scaffolding: the tree manifest is stale and the adoption guidance is too repetitive to be maximally useful for new users.

## Recommended Exact Edits
1. Update `docs/tree-manifest.txt` so it matches the tracked repository tree, including the `audits/` directory and current audit files.
2. Make `prompts/audit_repo.md` a thin redirect to `docs/strict-audit-prompt.md`, or remove one of the two to eliminate duplicate canonical audit guidance.
3. Rewrite `docs/tutorial.md` into one concrete end-to-end walkthrough with explicit stage outputs, instead of another adoption checklist.
4. Trim `examples/workflows/feature-workflow.md` and `examples/workflows/toolkit-adoption.md` to short pointers if `docs/tutorial.md` becomes the full walkthrough.
5. Expand `scripts/adopt-in-project.sh` so its name matches its behavior, or rename it to reflect that it only bootstraps `project_memory`.
6. Add stack-specific differentiators to `overlays/mobile-flutter/README.md`, `overlays/backend-node/README.md`, and `overlays/web-frontend/README.md`.
