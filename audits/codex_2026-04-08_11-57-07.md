# Codex Audit - 2026-04-08 11:57:07

## Executive Summary

The repository is close, but it is not yet public-release ready as a stack-agnostic foundation toolkit.

What is working:
- The root identity is stated consistently in `README.md`, `README_TH.md`, `AGENTS.md`, and `docs/architecture.md`.
- The overlay model is explicit and mostly well-scoped.
- The agent-team roles are clearly separated.
- The prompt pipeline has a single canonical source in `docs/prompt-pipeline.md`.
- The public repo scaffold is mostly present: license, contributing guide, code of conduct, security policy, PR template, issue templates, and CI all exist.

What is not working:
- The only worked example in the foundation examples is Python-specific, which violates the repo identity rule that stack-specific material belongs in overlays only.
- The repository duplicates canonical governance and lifecycle content in many places, which raises drift risk.
- Public-repo readiness checks are incomplete relative to the docs that define the repo's public adoption surface.

Summary counts:
- Critical: 1
- Medium: 4
- Minor: 3

## Critical Issues

### 1. Foundation example is Python-specific and should not live in `examples/`
- Location: `examples/worked_examples/python_service_feature.md:1-69`
- Severity: Critical
- Category: Foundation / Identity leakage
- Description: This file hardcodes a FastAPI scenario, router/service/repository layout, and Python verification examples. `docs/architecture.md:21-28` classifies `examples` as foundation content, so this file makes the foundation layer stack-specific.
- Impact: The repository reads as Python-first even though the stated identity is domain-agnostic and stack-agnostic at the root. That weakens adoption for non-Python users and violates the repo's own separation rule.
- Recommendation: Move this example out of the foundation tree and into the Python overlay, then add a stack-neutral worked example to the foundation examples directory.
- Suggested command: `extract` or `distill`

## Medium Issues

### 1. The canonical lifecycle is duplicated in too many places
- Location: `README.md:31-45`, `AGENTS.md:4-18`, `docs/prompt-pipeline.md:1-23`, `core/rules/output-contract.md`, `templates/runbooks/feature-flow.md`, `examples/workflows/feature-workflow.md`, `docs/real-world-integration.md:1-6`
- Severity: Medium
- Category: Docs / Prompt pipeline coherence
- Description: `docs/prompt-pipeline.md` is correctly positioned as the source of truth, but the full lifecycle and role order are restated in several other docs and templates.
- Impact: The content is currently consistent, but the repetition increases drift risk and makes future edits more expensive. One missed update will create conflicting guidance.
- Recommendation: Keep the full lifecycle only in `docs/prompt-pipeline.md`. Replace the duplicated blocks elsewhere with short references to the canonical file.
- Suggested command: `distill`

### 2. Security and ownership governance files are duplicated and already diverging
- Location: `SECURITY.md:1-5`, `.github/SECURITY.md:1-15`, `CODEOWNERS:1-4`, `.github/CODEOWNERS:1-7`
- Severity: Medium
- Category: Public repo hygiene / Docs drift
- Description: Both `SECURITY.md` files exist, but the root file is shorter and does not match the `.github` version. Both `CODEOWNERS` files exist, but the root file is only a comment plus a single catch-all rule while `.github/CODEOWNERS` has path-specific ownership.
- Impact: Future maintainers can easily edit one copy and forget the other. That creates an avoidable mismatch in a public repository where policy files are visible and expected to be authoritative.
- Recommendation: Pick one canonical location for each file and make the other a literal mirror or a one-line pointer. Do not let the contents diverge.
- Suggested command: `normalize`

### 3. Public repo checklist is incomplete relative to the actual adoption surface
- Location: `docs/public-repo-checklist.md:5-40`, `.github/workflows/release-check.yml:1-20`
- Severity: Medium
- Category: Public release readiness
- Description: The checklist verifies the basic legal and GitHub scaffold files, but it omits several docs that the README and tutorials treat as core adoption material, including `docs/agent-team-system.md`, `docs/real-world-integration.md`, `docs/repo-bootstrap.md`, `docs/release-process.md`, and `docs/codex-review-prompt.md`.
- Impact: A release can pass checklist-oriented review while still missing the documentation a new adopter actually needs. The release-check workflow also verifies only a subset of the checklist, so the two sources of truth are not aligned.
- Recommendation: Expand the checklist to include the full public-facing doc set, then make `release-check.yml` enforce the same exact list.
- Suggested command: `harden`

### 4. The foundation examples have no stack-neutral worked example
- Location: `examples/worked_examples/python_service_feature.md:1-69`
- Severity: Medium
- Category: Adoption guidance
- Description: The only worked example is Python-specific. There is no foundation-level example that demonstrates the lifecycle without a stack assumption.
- Impact: New adopters do not get a neutral example of how to apply the toolkit before choosing an overlay. That makes the foundation feel thinner than it should and increases the risk that users copy stack-specific patterns into the wrong layer.
- Recommendation: Add a stack-neutral worked example in `examples/worked_examples/` and keep stack-specific versions under the relevant overlay tree.
- Suggested command: `onboard`

- Location: `README.md:31-59`
- Severity: Minor
- Category: Docs maintenance
- Description: The README re-states the lifecycle, team model, and included folder list that are already defined in the dedicated docs.

- Suggested command: `distill`
