## 1. Executive Summary

The repository has a coherent identity and a sound architecture. The foundation/overlay boundary is philosophically clear, the canonical lifecycle is explicit and well-referenced, and the 7 core lifecycle prompts and 9 skill contracts are solid. The governance layer (`AGENTS.md`, `core/`, role system) is consistent with itself.

Since the prior Claude audit (123118), three issues were resolved: the CI broken path for the Python example was fixed, `docs/push-guide.md` was moved to `scripts/push-guide.md`, and CHANGELOG received an explanatory note about same-day versioning. That progress is real.

What remains: there are no release blockers in the strict sense (CI will pass, all required files exist, no private URLs are embedded in public paths), but there are **five medium issues** that damage credibility on first inspection by a serious adopter, and **seven minor issues** that produce friction or inconsistency.

The most significant ongoing deficiencies are:
1. `scripts/codex-final-review-prompt.txt` is still a one-line stub redirect referenced by the tutorials — a dead resource.
2. Three of four overlays (mobile-flutter, backend-node, web-frontend) have nearly identical `AGENTS.overlay.md` files that provide almost no stack-specific value beyond the foundation.
3. Three of four overlay worked examples are 10–13 lines — not usable as adoption evidence.
4. "Codex" is used as a product name in foundation-level file names and content, violating the stack-agnostic rule.
5. `docs/repo-bootstrap.md` omits `templates/project_memory/patterns.md` from its bootstrap list while every other adoption doc includes it.

**Verdict: Almost ready.** Fixable in one focused pass. None of the remaining issues require architectural changes.

### M2 — Three overlays have near-identical `AGENTS.overlay.md` files with no stack-specific content

**Files affected:**
- `overlays/mobile-flutter/AGENTS.overlay.md`
- `overlays/backend-node/AGENTS.overlay.md`
- `overlays/web-frontend/AGENTS.overlay.md`

**Problem:** All three have the same four "Boundary rules":
```
- Keep stack entry concerns in the stack entry layer.
- Keep business rules out of presentation/transport glue.
- Isolate external providers and side effects.
- Keep project-specific rules in the consuming repo.
```

These rules are indistinguishable from general foundation principles. They say nothing specific to Flutter, Node, or web frontends. Contrast with `overlays/python-service/AGENTS.overlay.md` which specifies: route handlers stay thin, domain logic stays framework-light, persistence lives in repositories, workers stay out of route handlers.

An adopter picking up the mobile-flutter overlay gets rules they already had from the foundation. The overlay provides no additive guidance.

**Fix for each:**

`overlays/mobile-flutter/AGENTS.overlay.md` — replace the generic boundary rules with Flutter-specific ones, e.g.:
```
## Boundary rules
- Presentation widgets must not contain business logic or direct provider calls.
- Use cases or interactors must not import Flutter widgets.
- Repository implementations must not leak Flutter-specific types into domain interfaces.
- BLoC/Cubit/Provider state must not call external APIs directly — route through services or use cases.
- Navigation logic must not be embedded in widget trees.
```

`overlays/backend-node/AGENTS.overlay.md` — add Node-specific specificity, e.g.:
```
## Boundary rules
- Route handlers (Express/Fastify/etc.) must not contain business logic.
- Services must not import Express/Fastify request or response types.
- Repositories must not contain validation or transformation logic.
- Environment and config must not be read directly inside service or domain layers.
- Middleware must not perform business orchestration.
```

`overlays/web-frontend/AGENTS.overlay.md` — add frontend specificity, e.g.:
```
## Boundary rules
- Page components must not contain reusable UI logic — that belongs in feature or component layers.
- Data fetching must not be embedded directly in reusable presentational components.
- Global state must not be mutated from within pure presentational components.
- Design system primitives must not contain business or feature logic.
- API calls must go through a service or data layer, not directly from component files.
```

### m4 — `scripts/adopt-in-project.sh` will fail silently if run without argument substitution

**File:** `scripts/adopt-in-project.sh`

**Problem:** Line 3:
```bash
TOOLKIT_URL=${2:-"<your-toolkit-repo-url>"}
```

If run without a second argument, the script passes the literal string `<your-toolkit-repo-url>` to `git submodule add`. This will produce a confusing git error like "Repository not found" rather than a helpful message indicating the URL was not configured. The script echo before running says "Adding toolkit as submodule..." which implies it will succeed.

**Fix:** Add an explicit guard:
```bash
TOOLKIT_URL=${2:-}
if [ -z "$TOOLKIT_URL" ]; then
  echo "Error: provide the toolkit repo URL as the second argument." >&2
  echo "Usage: $0 [target-dir] <toolkit-repo-url>" >&2
  exit 1
fi
```
