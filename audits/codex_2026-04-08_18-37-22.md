# Public Foundation Toolkit Audit

## 1. Executive Summary
- Total issues found: 0 critical, 3 medium, 2 minor.
- The repository is materially cleaner than the earlier snapshot: the tree manifest is synchronized, root onboarding is mostly stack-agnostic, and the public-release gate now checks for path drift and internal-reference leakage.
- The remaining gaps are public-facing rather than structural: one common-start tutorial still names Flutter in a foundation path, one public prompt still uses internal-only release-gate framing, and the top-level start-here page is still too abstract for first-time adoption.

## 2. Critical Issues
- None found in the current state of the repository.

## 3. Medium Issues
- `docs/tutorials/00-common-start.md:101-107` and `docs/tutorials/00-common-start_EN.md:101-107` still tell readers to choose `mobile-flutter` when the platform is known. That is a stack-specific example inside the common blank-folder bootstrap guide, so it reintroduces platform bias into a foundation tutorial. Replace it with neutral overlay-selection guidance or move stack examples into overlay-specific docs.
- `prompts/review/internal_release_gate.md:1-4` is stored in the public `prompts/review` tree even though the prompt explicitly says to review the repository as an internal toolkit, not a public marketing repo. That is internal-only framing exposed in the public surface. Move it under `prompts/internal/` or rewrite it as a public release-gate prompt.
- `README_START_HERE.md:3-8` still says “Run Lead Prompt / Architecture Prompt / Feature Prompt” without naming the actual files or the canonical prompt hub. That makes the main entry point vague for a public repo and weakens adoption guidance. Link the quick-start steps to the exact docs or prompt files.

## 4. Minor Issues
- `docs/tutorials/index.md:17-36` and `docs/tutorials/index_EN.md:17-36` duplicate the entire bilingual tutorial matrix. The content is correct, but this is a maintenance drift risk because every tutorial addition or rename has to be mirrored twice.
- `README.md:80-101` and `docs/how-to-use.md:48-118` repeat overlapping adoption guidance. The duplication is not a correctness bug, but it makes the public narrative longer than necessary and increases the chance of future inconsistency.

- The root docs themselves are now mostly clean; the remaining violations are concentrated in the common-start tutorial and the review prompt tree.

## 7. Public Repo Readiness Assessment
- The repository is close to public-release shape, but it is not fully release-clean yet.
- The manifest and release gate are now meaningful, which is a real improvement.
- The remaining issues are enough to keep the repo from being labeled production-ready for a public foundation release because they affect the public contract and first-run onboarding.

## 8. Exact Recommended Edits
- Rewrite `docs/tutorials/00-common-start.md` and `docs/tutorials/00-common-start_EN.md` so the common start path stays neutral and only says to choose an overlay if the stack is known.
- Move `prompts/review/internal_release_gate.md` into `prompts/internal/` or rename/reframe it so it no longer presents internal-only language on a public prompt path.
- Replace the quick-start bullets in `README_START_HERE.md` with exact links to the canonical prompt docs or tutorial pages.
- Reduce duplication between `README.md` and `docs/how-to-use.md` by letting one page own the adoption narrative and turning the other into a shorter pointer.
- Shorten one of the bilingual tutorial hubs, or make one file the canonical list and the other a minimal mirror, to reduce drift risk.
