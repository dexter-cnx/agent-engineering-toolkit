# Repository Audit: agent-engineering-toolkit

**Auditor:** Claude Sonnet 4.6  
**Date:** 2026-04-08 09:31:46  
**Scope:** Public, production-ready foundation toolkit audit  
**Constraints from AGENTS.md:** Root must remain stack-agnostic. Stack-specific assumptions must live only in overlays.

---

## 1. Executive Summary

The structural concept is sound: a domain-agnostic foundation layer with stack-specific specialization in overlays, driven by a 7-stage lifecycle and a 7-role team model. AGENTS.md, the architecture rationale, and the docs discipline are coherent.

However, the actual content is consistently skeletal. Prompts are 10–20-line instruction stubs with no substance. Skills are 2–4-line purpose statements with no implementation. Overlays describe what to add but contain nothing. Templates are empty forms. The public repository hygiene is incomplete: there is no `.github/` directory, no CI workflows, no CODEOWNERS, no SECURITY.md, and a private GitHub handle is hardcoded in two shipped files. The repo self-declares as "production-ready" in its own README, which is not accurate at this point.

## 2. Critical Issues

### C1 — No `.github/` directory exists
README.md and README_TH.md both list `.github/` in the repository structure tree. The public-repo-checklist explicitly requires CI workflows, PR templates, issue templates, and CODEOWNERS. None of these files exist. The directory does not exist. This is a public release blocker.

### C2 — Hardcoded private GitHub handle in shipped files
`scripts/push-new-repo.sh` and `docs/push-guide.md` both hardcode `git@github.com:dexter-cnx/agent-engineering-toolkit.git`. This is a private identifier. It must not ship in a public toolkit. Any consumer who runs the script without reading it will point at someone else's repo.

### C3 — LICENSE has no copyright holder
`LICENSE` reads `Copyright (c) 2026` with no name or organization. An MIT license without a named holder is legally incomplete. This blocks public release.

### C4 — Duplicate docs with inconsistent cross-references
Two parallel pairs exist:
- `docs/how-to-use.md` (short, 12 sections) and `docs/how_to_use.md` (long, 15 sections)
- `docs/how-to-use_TH.md` (short Thai) and `docs/how_to_use_TH.md` (long Thai)

README references `docs/how-to-use.md`. The tutorial files reference `docs/how_to_use.md`. The short and long files have different content and are not labeled as such. A new user cannot know which to read. This is structural duplication with no rationale.

### C5 — CODEOWNERS does not exist
Referenced in README ("review `CODEOWNERS`"), in the public-repo-checklist ("replace placeholder GitHub handles in `CODEOWNERS`"), and in push-guide.md ("update CODEOWNERS if more maintainers are added"). The file has never been created.

### C6 — No CI workflows
The release-process doc says "Run CI." The public-repo-checklist says "verify CI passes." README references `.github/` with an implied CI. No `.github/workflows/` directory exists. No CI definition exists.

### C7 — README pre-publish checklist left in main README
`README.md` lines 115–126 contain a "Before publishing broadly" checklist that includes "confirm your preferred license choice" and "review `CODEOWNERS`" — items that are not complete. This checklist appears verbatim in the public-facing README, telling external readers the repo is not yet ready. It should be removed from README and moved to `docs/public-repo-checklist.md` (where it partially overlaps already).

## 3. Medium Issues

### M1 — Prompts are stubs, not production prompts
Every prompt in `prompts/` is 10–20 lines. They name a role, list a task, and name expected outputs. They have no prompt engineering: no context injection guidance, no output format specs, no quality bars, no example outputs, no handling of missing context. `plan_change.md` is 18 lines. `implement_change.md` is 15 lines. These are outlines for prompts, not prompts. A consuming team cannot use these directly with a model and expect structured, reliable output.

### M2 — Skills are purpose statements only
Every file in `skills/*/README.md` is 2–5 lines. `skills/repo-audit/checklist.md` is 6 bullet points. `skills/architecture-review/README.md` is one sentence. None of the skills contain: trigger conditions, step-by-step execution, input/output contracts, example usage, or integration with adjacent skills. The skill-router in `core/routing/skill-router.md` maps task types to skills, but the skills themselves contain nothing to execute.

### M3 — Overlays are outlines, not usable extensions
Each overlay is two files (~10–14 lines each). They name what should be added ("clean architecture guidance for app layers / UI/state/navigation conventions / localization rules") but contain none of it. An engineer who adopts `mobile-flutter` gets 14 lines pointing at concepts, not an actual extension of the foundation. The overlays state their own rule at the end: "This overlay extends the foundation. It should not redefine the foundation's identity." But they don't extend anything.

### M4 — Templates are empty forms
`templates/review/review-template.md`: 5 headers, no guidance.  
`templates/verification/verification-template.md`: 4 headers, no guidance.  
`templates/runbooks/feature-flow.md`: 7 numbered lines.  
`templates/project-bootstrap/README_BOOTSTRAP.md`: 10 lines restating what to copy.  
None contain example content, fill-in guidance, or quality criteria. A team copying these gets empty markdown files.

### M5 — Agent team role files provide no behavioral guidance
Each file in `agent_team/` is 5–7 lines of responsibility bullets. No prompt language. No constraints. No interaction patterns. No difference in format from the `docs/agent-team-system.md` that describes the same roles in more words. The individual role files add no operational value beyond the summary doc.

### M6 — SECURITY.md is absent
`docs/codex-review-prompt.md` (the toolkit's own audit prompt) explicitly lists SECURITY.md as a required public repo hygiene file. It is missing.

### M7 — PR template and issue templates are absent
Also explicitly listed in the toolkit's own codex-review-prompt as required. Missing.

### M8 — `scripts/adopt-in-project.sh` is not a script
The file contains two `echo` statements and no actual logic. It does not install, copy, or configure anything. A user who runs it gets two printed lines.

### M9 — `scripts/codex-final-review-prompt.txt` duplicates `docs/codex-review-prompt.md`
Both files contain the same audit prompt text. One is plain text, one is markdown with a wrapper. There is no clear reason both exist. One should reference or supersede the other.

### M10 — `project_memory/` (live toolkit memory) is nearly empty
`decisions.md` has 2 bullet points. `known_constraints.md` has 1 bullet point. AGENTS.md defines memory as non-optional. The toolkit's own memory is not being maintained by its own rules.

### M11 — "OpenClaw" reference is unexplained
`docs/how-to-use.md`, `docs/how-to-use_TH.md`, `docs/real-world-integration.md`, and the Thai variants all reference "OpenClaw" as a supported AI coding tool. This term is not explained, not linked, and may be a private tool or internal name. External readers have no context for it.

### M12 — `examples/` provides no real adoption evidence
Four files total. `feature-workflow.md` is 9 lines. `toolkit-adoption.md` is 6 lines. `example_feature_request.md` is 12 lines. These examples describe the process at the same abstraction level as the docs themselves. There are no end-to-end worked examples, no sample prompt inputs/outputs, no before/after artifacts.

## 4. Minor Issues

### m1 — File naming convention is inconsistent
Hyphen (`how-to-use.md`) and underscore (`how_to_use.md`) coexist for the same conceptual document. AGENTS.md documentation rules reference `docs/how-to-use(.md / _TH.md)`. Standardize to one convention.

### m2 — README_TH.md omits `docs/codex-review-prompt.md` from its doc list
English README lists it; Thai README does not. The Thai version is incomplete by comparison.

### m3 — `CODE_OF_CONDUCT.md` has no enforcement contact
Standard practice for public repos is to provide a contact (email or issue link) for CoC violations. This file has none. Also, it's minimal enough that adopting the Contributor Covenant (with proper attribution) would be more credible.

### m4 — `CHANGELOG.md` has no "Unreleased" section and no format guidance
Only one entry. No standard format for future contributors to follow.

### m5 — `docs/roadmap.md` is all aspiration with no specifics
"Later: dashboard and execution telemetry" gives no timescale, owner, or design rationale. Readers cannot assess feasibility or direction.

### m6 — `docs/real-world-integration.md` is 22 lines of bullet points
It covers Codex, Claude Code, OpenClaw, CI/CD, and submodules in fewer than 25 lines total. This is a critical adoption doc that should be substantially more detailed.

### m7 — No `.gitignore` in repository
A public toolkit repo should define what not to commit, even if it's primarily markdown.

### m8 — Example feature uses UI-specific language at foundation level
`docs/how-to-use.md` uses "billing settings page" as the example feature. The Thai `how-to-use_TH.md` uses "หน้าตั้งค่า billing" (billing settings screen). This frames the primary usage example around UI/mobile — a mild foundation leak.

### m9 — `docs/release-v0.1.0.md` is an internal document in a public repo
Not harmful, but adds noise for external consumers who will wonder what "release candidate" means after they've already cloned it.

## 6. Foundation vs Overlay Violations

**No stack-specific code bleeds into the root.** The separation of concepts is maintained. This is the strongest aspect of the repo.

However:

**V1 — Overlays contain no stack-specific content.** They describe what stack-specific content should be added ("clean architecture guidance for app layers") but contain none of it. The boundary is honored conceptually but the overlays are empty on their side of the boundary. A team consuming `mobile-flutter` gets a 14-line file that says "you should add localization rules" — not localization rules.

**V2 — Foundation docs use UI-framed examples.** The primary feature example across `docs/how-to-use.md`, the Thai guide, and the tutorial is "billing settings page/screen." This is a minor but real foundation-level framing toward UI/app projects rather than staying agnostic.

**V3 — `docs/real-world-integration.md` mentions Claude Code and Codex by name at foundation level.** These are acceptable since they are the primary consumers. OpenClaw is unexplained.

## 9. Final Verdict

**NOT READY**

The architectural concept is solid and the foundation-vs-overlay discipline is well-reasoned. AGENTS.md is the strongest file in the repo. The dual-language commitment is valuable.

But:
- Six public hygiene requirements are completely absent (CI, CODEOWNERS, SECURITY.md, PR template, issue templates, complete LICENSE).
- A private GitHub handle ships in two files.
- README contains an incomplete pre-publish checklist visible to all readers.
- The prompts, skills, overlays, templates, and examples are consistently skeletal — the repo self-describes as "production-ready" but the content would not survive a first real adoption attempt without a team significantly extending the stubs themselves.
- The toolkit's own memory rules are not followed in its own `project_memory/`.

The gap between the stated purpose ("production-ready foundation toolkit") and the actual depth of the content is too large for a public release. Fix the public hygiene failures first (Edits 1–8 above), then invest in substance: at least one worked example, at least one fleshed-out overlay, and prompts with enough content to produce structured output when fed to a model.
