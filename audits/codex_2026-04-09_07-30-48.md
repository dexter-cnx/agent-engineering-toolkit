# Repository Audit

Date: 2026-04-09 07:30:48
Scope: public, production-ready audit of the foundation toolkit, with attention to repository identity, overlays, docs, prompts, templates, examples, and public-release hygiene.

## Executive Summary

This repository is **almost ready** for public release as a foundation toolkit.

The public-release gate now passes, the root-level Flutter pack files are no longer present, and the append-manifest helper has also been removed from the working tree. That resolves the previously critical release-blocking drift.

The only tracked working-tree change at the moment is `docs/tree-manifest.txt`, so the manifest itself should be treated as the active review target before the next release cut.

What remains is mostly presentation and adoption clarity:
- the root docs still over-center the Mobile Flutter overlay
- the overlay catalog count is repeated more often than necessary
- the Flutter overlay helper docs are very thin

Those issues are real, but they are not release blockers at this point.

## Critical Issues

None found in the current state.

## Medium Issues

1. The foundation navigation is still Flutter-biased
- `README.md`, `README_TH.md`, `README_START_HERE.md`, `docs/how-to-use.md`, `docs/how-to-use_TH.md`, `docs/tutorial.md`, `docs/tutorial_TH.md`, `docs/overlays.md`, `docs/tutorials/index.md`, and `docs/tutorials/index_EN.md` still center the Mobile Flutter overlay first.
- Evidence: `README.md:25-69`, `README_TH.md:23-38`, `README_START_HERE.md:9-20`, `docs/how-to-use.md:12-43`, `docs/how-to-use_TH.md:12-41`, `docs/tutorial.md:14-25`, `docs/tutorial_TH.md:14-42`, `docs/overlays.md:12-25`, `docs/tutorials/index.md:3-17`, `docs/tutorials/index_EN.md:3-17`
- Impact: the repo still reads more like a Flutter-first toolkit with other overlays attached than a neutral multi-overlay foundation. That weakens adoption clarity for backend-node, python-service, and web-frontend users.

2. The overlay catalog count is duplicated too widely
- The “23 skills” count appears in several public docs instead of in one overlay-owned place.
- Evidence: `README.md:25-27`, `README_TH.md:23-25`, `docs/how-to-use.md:35-43`, `docs/how-to-use_TH.md:35-41`, `docs/overlays.md:12-25`
- Impact: every future catalog change will require synchronized edits in multiple foundation docs, which creates avoidable drift risk.

3. The overlay-level `HOW_TO_APPLY.md` and `VALIDATION.md` are extremely thin
- These files exist under `overlays/mobile-flutter/`, but they are minimal and easy to miss.
- Evidence: `overlays/mobile-flutter/HOW_TO_APPLY.md:1-6`, `overlays/mobile-flutter/VALIDATION.md:1-6`
- Impact: if retained, they need clearer naming or stronger guidance about their limited purpose. As written, they are easy to confuse with packaging artifacts rather than true overlay docs.

## Minor Issues

1. The tutorial hubs are coherent but not neutral
- `docs/tutorials/index.md` and `docs/tutorials/index_EN.md` still feel like a Flutter launchpad instead of a neutral cross-overlay index.
- Evidence: `docs/tutorials/index.md:3-17`, `docs/tutorials/index_EN.md:3-17`
- Impact: this is not a blocker, but it reinforces the same identity imbalance seen in the root docs.

2. The root README family still spends a lot of its visible space on Flutter
- `README.md` and `README_TH.md` both include a prominent Mobile Flutter status section near the top.
- Impact: this is acceptable as a visible overlay section, but it is still more dominant than ideal for a public foundation repo.
