# Repository Audit

Date: 2026-04-09 06:49:33
Scope: public, production-ready audit of the foundation toolkit, with special attention to root stack-agnosticity, overlays, docs, prompts, skills, templates, examples, and public-release hygiene.

## Executive Summary

This repository is close to a usable public foundation toolkit, but it is not ready for a public release as-is.

The biggest blockers are:
- the authoritative tree manifest is stale and does not match the real repository tree
- root-level files still carry a Mobile Flutter pack identity
- the public-facing navigation is overly centered on Mobile Flutter instead of presenting the foundation and overlays neutrally

The public docs and prompt structure are otherwise coherent, and the main link set I checked resolved correctly. The overlay skill folders also appear structurally complete. The release blockers are identity and manifest consistency, not broken navigation.

## Critical Issues

1. `docs/tree-manifest.txt` is no longer authoritative
- The public-release gate expects the manifest to match the actual repository tree, but `scripts/check-public-repo.sh` fails with tree drift.
- Evidence: `scripts/check-public-repo.sh:32-39`, `scripts/check-public-repo.sh:109-113`
- Evidence: `docs/public-repo-checklist.md:55-63`
- The current manifest is missing root files that exist in the repository, including `HOW_TO_APPLY.md` and `VALIDATION.md`, and it also omits the full `overlays/mobile-flutter/skills/**` subtree.
- Impact: the repo cannot honestly claim that its machine-readable tree snapshot is current, and the public-release gate becomes unreliable.

2. Root-level stack-specific artifacts violate the foundation identity
- `HOW_TO_APPLY.md` is a Mobile Flutter pack at the repository root, not in an overlay.
- Evidence: `HOW_TO_APPLY.md:1-10`
- `VALIDATION.md` is a root-level validation report for that same pack.
- Evidence: `VALIDATION.md:1-8`
- The root contract says stack-specific assumptions must live in overlays, and the root must stay stack-agnostic.
- Evidence: `AGENTS.md` instructions in this repo, plus `docs/architecture.md:6-18`
- Impact: public consumers can see a root-level Flutter-specific package artifact and validation report, which directly contradicts the stated foundation identity.

## Medium Issues

1. The foundation navigation is still Flutter-biased
- `README.md`, `README_TH.md`, `README_START_HERE.md`, `docs/tutorial.md`, `docs/how-to-use.md`, `docs/overlays.md`, `docs/tutorials/index.md`, and `docs/tutorials/index_EN.md` all steer readers toward the Mobile Flutter overlay first.
- Evidence: `README.md:25-69`, `README_TH.md:23-38`, `README_START_HERE.md:9-20`, `docs/how-to-use.md:12-43`, `docs/tutorial.md:14-25`, `docs/overlays.md:12-25`, `docs/tutorials/index.md:3-17`, `docs/tutorials/index_EN.md:3-17`
- Impact: the public foundation reads like a Flutter-first toolkit even though the repo explicitly claims to be domain-agnostic and stack-agnostic at the root. That weakens discoverability for backend-node, python-service, and web-frontend users.

2. The same overlay count and catalog summary are duplicated across too many docs
- The hardcoded “23 skills” / “23 reusable skills” wording appears in multiple public docs, not just the overlay itself.
- Evidence: `README.md:25-27`, `README_TH.md:23-25`, `docs/how-to-use.md:35-43`, `docs/overlays.md:12-25`
- Impact: every catalog change now requires synchronized edits in several foundation docs. That creates drift risk and makes the documentation more brittle than it needs to be.

## Minor Issues

1. `docs/tutorials/index.md` and `docs/tutorials/index_EN.md` are useful hubs, but they still behave like a Flutter launchpad instead of a neutral cross-overlay tutorial index
- Evidence: `docs/tutorials/index.md:3-17`, `docs/tutorials/index_EN.md:3-17`
- Impact: not a blocker, but it further reinforces the Flutter-first feel and could confuse new adopters who came here for the foundation, not one overlay.

## Foundation vs Overlay Violations

- `HOW_TO_APPLY.md` is a root-level Mobile Flutter pack artifact. That belongs in an overlay or an external consuming repo, not in the foundation root.
- `VALIDATION.md` is a root-level validation report for that same pack. That also belongs outside the foundation root.
- `README.md`, `README_TH.md`, `README_START_HERE.md`, `docs/how-to-use.md`, `docs/tutorial.md`, `docs/overlays.md`, and the tutorial hub pages all make Mobile Flutter feel like the default path instead of one overlay among several.
- `docs/tree-manifest.txt` is also a boundary problem because it is meant to be the authoritative public snapshot, but it is missing overlay skill subtrees and root artifacts that exist in the repo.

## Public Repo Readiness Assessment

Current status: not ready.

Why:
- the repository fails its own public-release gate because the tree manifest is stale
- root-level stack-specific artifacts contradict the foundation identity
- onboarding and tutorial navigation are not yet neutral enough for a public foundation toolkit

What is already in decent shape:
- public hygiene files are present
- the public docs structure is coherent
- the markdown links I spot-checked resolved
- the overlay skill folders appear structurally complete

## Exact Recommended Edits

1. Refresh `docs/tree-manifest.txt`
- Regenerate it from the actual repository tree.
- Include the current root files and the `overlays/mobile-flutter/skills/**` subtree.
- If `HOW_TO_APPLY.md` and `VALIDATION.md` are not meant to be public artifacts, remove them instead of silently leaving them out of the manifest.

2. Move or remove root-level Flutter pack artifacts
- Move `HOW_TO_APPLY.md` and `VALIDATION.md` into an overlay-specific location if they are still useful.
- If they are obsolete, delete them.
- Do not leave stack-specific pack files at the foundation root.

3. Reduce Flutter bias in the foundation entry points
- Update `README.md`, `README_TH.md`, and `README_START_HERE.md` so they present the foundation and all overlays more evenly.
- Keep Mobile Flutter as one option, not the first and only visibly detailed path.
- Consider a short overlay chooser section rather than a Flutter-heavy quick path.

4. Consolidate the skill-count wording
- Remove repeated “23 skills” copy from foundation docs where it is not essential.
- Keep the catalog count, if needed, in one overlay-owned place only.

5. Rebalance the tutorial hub
- Expand `docs/tutorials/index.md` and `docs/tutorials/index_EN.md` so they clearly surface the non-mobile overlays as well.
- If the intent is to keep Flutter first, say that explicitly and explain why; otherwise make the hub neutral.
