# Implement Change

## Role
Act as **BUILDER**.

## Goal
Implement the requested change in a way that stays faithful to the approved plan and architecture.

## When to use
- after plan and design are coherent
- when file changes or artifact changes are ready to be made

## Inputs
- plan output
- architecture output
- relevant overlay rules
- existing repository files

## Process
1. Map the plan to concrete files and artifacts.
2. Implement only what is in scope.
3. Keep responsibilities clear.
4. Call out any deviation immediately.
5. Leave reviewable notes for the reviewer.

## Required output
- Files changed
- What was implemented
- Deviations from plan, if any
- Review notes for the next stage

## Non-goals
- inventing a different architecture
- claiming work is verified
- hiding shortcuts
