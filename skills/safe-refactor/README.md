# Safe Refactor

## Purpose
Improve structure without changing intended behavior.

## When to use
- Naming cleanup
- Responsibility cleanup
- Boundary improvement

## When not to use
- New feature delivery masquerading as refactor

## Inputs
- Current structure
- Intended preserved behavior
- Risk areas

## Outputs
- Refactor plan
- Safety notes
- Behavior-preservation cautions

## Non-goals
- Feature expansion
- Silent architectural redesign
